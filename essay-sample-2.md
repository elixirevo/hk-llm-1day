### 1. 인젠트 지원동기

```
삼성전자 네트워크사업부 연계 프로젝트에서 400명이 사용하는 시스템을 개발하며, PostgreSQL을 실행 계획을 직접 제어해야 하는 레이어로 다뤘습니다. JSON을 JSONB로 전환하고 GIN 인덱스를 적용했으며, LIMIT 파라미터가 Generic Plan을 유도하는 문제를 실행계획 분석으로 확인한 뒤 상수로 변경해 Top-N Sort를 유도했습니다. 그 결과 API 응답 속도를 4.21초에서 0.123초까지 단축했습니다.

이 과정에서 성능 개선의 핵심은 애플리케이션 코드가 아니라 DB 내부 동작 이해에 있다는 점을 확인했습니다. 이후 기능 구현보다 실행계획, 인덱스 구조, 트랜잭션 흐름을 분석하는 작업에 더 많은 시간을 쓰게 됐고, DB를 사용하는 개발자가 아니라 DB 동작을 책임지는 역할로 방향을 잡았습니다.

인젠트를 선택한 것은 PostgreSQL을 내부 서비스에 활용하는 수준이 아니라, PostgreSQL 자체의 설치·운영·장애 대응을 전문적으로 수행하는 사업 구조이기 때문입니다. 국내에서 PostgreSQL 국제 커뮤니티에 Professional Service Provider로 등록된 회사로서, 고객 환경에서 DB 엔진 동작을 직접 분석하고 판단하는 경험을 쌓을 수 있는 곳이라고 판단했습니다.

쿼리 실행계획을 분석해 병목을 제거했던 경험을, 이제는 고객사의 PostgreSQL 운영 환경 전체를 진단하고 개선하는 역할로 확장하고 싶어 지원했습니다.
```

### 2. 실패 경험 / 극복 사례

```
[40시간짜리 작업을 1시간으로 줄인 판단]
삼성전자 네트워크사업부 연계 프로젝트 초기 요구사항은 엑셀 기반 협업을 웹 UI로 옮기는 것이었습니다. 해당 프로젝트는 단순 반복 작업을 자동화해 설계-해석 프로세스 전반의 효율을 높이는 것이 목표였습니다.

하지만 이후 실무자 면담을 진행하면서 문제 정의부터 잘못되었다는 것을 확인하였습니다. 각 설계팀은 VISIO와 EXCEL로 부품 배치를 수행하고, 열 해석 담당자가 메일과 구두로 이를 받아 다시 정리했습니다. 설계 변경이 생기면 취합과 정리가 처음부터 반복되었고, 변경 이력은 어디에도 남지 않았습니다. 열 해석 모델링 준비에만 약 40시간이 소요되는 구조였습니다.

문제는 도구가 아니라 협업 방식 자체였습니다. 웹 UI를 추가하는 것으로는 이 반복 구조가 없어지지 않는다고 판단해, 설계-해석 흐름 전체를 단일 시스템으로 통합하는 방향으로 설계를 전면 재작성했습니다. Git의 버전 관리 개념을 차용해 작업 상태를 버전 단위로 분리했고, 다중 사용자 환경에서 충돌이 발생하지 않는 구조를 ERD부터 직접 설계했습니다. 실무자가 "어떤 버전이 현재 기준인지"를 혼동 없이 확인할 수 있도록 상태 전이 흐름을 설계 단계에서 정의했습니다.

결과적으로 열 설계와 해석을 통합된 환경에서 수행할 수 있는 플랫폼을 개발하였습니다. 모델링 준비 시간은 40시간에서 1시간으로 단축되었고, 400명 이상의 실무자 공청회를 거쳐 시스템을 성공적으로 런칭하였습니다. 이후 삼성전자 연계 프로젝트 발표회에서 그 공로를 인정받아 우수상을 받았습니다.

이 경험 이후 저는 요구사항을 받으면 "왜 이 문제가 반복되고 있는가"를 먼저 확인합니다. 구현 속도보다 방향이 맞는지가 먼저라는 판단은 이 프로젝트에서 결과로 확인하였습니다.
```

### 3. 팀워크 경험

```
[코드보다 먼저 정리한 기준]
TMI 프로젝트에서 6명의 팀원과 함께 개발을 진행했습니다. 개발 초반, 저는 API 연동 과정에서 반복되는 문제가 기술 수준이 아니라 팀 기준의 부재에서 온다는 것을 확인했습니다.

당시 팀 내 API들은 단순 검증 실패조차 Exception으로 처리하고 있었습니다. 존재하지 않는 리소스 조회, 비즈니스 로직 실패, 시스템 오류가 모두 @RestControllerAdvice 하나로 흘러들었습니다. 정상적인 흐름 제어까지 예외 기반으로 처리되면서, 스택 트레이스 생성과 예외 전파 과정이 반복되었습니다. 특히 예외 객체 생성과 스택 정보 수집 비용이 매 요청 발생하며 단순 조회 실패조차 시스템 오류 처리 경로를 타는 성능 비효율 문제가 있었습니다.

저는 이 문제의 원인이 "예외를 너무 많이 쓰는 것"이 아니라 "실패와 오류를 구분하지 않은 구조"에 있다고 판단했습니다. 비즈니스 로직 실패와 시스템 예외를 분리하는 ServiceResult<t> 응답 구조를 직접 설계해 제안했습니다. 성공 흐름은 ServiceResult.ok(data), 실패 흐름은 ServiceResult.fail(ErrorCode)로 분기하고, 사용자 정의 ErrorCode를 팀 공통 규칙으로 정의했습니다. 응답 규칙, 로깅 기준, 신규 API 추가 시 확장 방법까지 Notion에 문서화해 팀 전체가 같은 기준으로 작업할 수 있도록 했습니다.

SSE 기반 실시간 알림을 설계할 때도 같은 방식으로 접근했습니다. 알림 생성 로직을 비즈니스 로직에서 분리해 도메인 이벤트 발행 이후 이벤트 리스너에서 처리하는 흐름으로 구현했고, 팀원이 새로운 이벤트 유형을 추가할 때 기존 코드를 수정하지 않아도 되는 구조를 만들었습니다. 이 역시 설계 의도를 문서로 정리해 온보딩 시간을 줄였습니다.

결과적으로 API 형식이 통일되면서 프론트엔드 연동 이슈가 줄었고, 이 프로젝트는 SSAFY 공통 프로젝트 우수상을 받았습니다. 팀에서 기준을 먼저 정리하는 사람이 실제 개발 속도와 코드 품질 양쪽에 영향을 준다는 점을 이 프로젝트에서 직접 확인했습니다.
```

### 4. 직무 관련 프로젝트 / 연구 경험

```
[세 프로젝트, 일관된 방향]
백엔드 개발자로서 세 개의 실제 서비스 개발 프로젝트를 수행했습니다. 세 프로젝트에서 공통으로 유지한 방향은 하나입니다. 기능을 추가하기 전에 운영 중에 발생할 문제가 있는지 꼼꼼하게 점검하는 것입니다.

삼성전자 네트워크사업부 연계 프로젝트에서 PCB 설계 협업 시스템을 개발했습니다. 백엔드 70%를 담당하며 REST API 설계, 버전 관리 ERD 설계, 일정 관리를 맡았습니다. 수천 개의 부품 데이터 검색에서 응답이 4.21초에 달하는 병목이 발생했습니다. JSON 컬럼은 인덱싱이 불가능해 Seq Scan이 반복됐고, JSONB 전환 + GIN 인덱스로 구조를 바꿨습니다. EXPLAIN ANALYZE로 실행계획을 확인하던 중 LIMIT 파라미터가 Generic Plan을 유발한다는 점을 발견해 상수 LIMIT(5)로 변경, Top-N Sort가 적용되도록 유도했습니다. 응답은 0.123초로 개선됐습니다. 다중 사용자 충돌 문제는 버전 단위 상태 분리 구조로 설계 단계에서 차단했습니다. 결과적으로 이 프로젝트는 우수상을 받았고, 400명 이상 실무자 공청회를 거쳐 현장에서 실제로 사용되는 성과를 이뤘습니다.

TMI 프로젝트에서는 인기 게시글 랭킹 산정, SSE 기반 실시간 알림, 예외 응답 구조 개선을 담당했습니다. Spring Scheduler로 10분 주기 랭킹 재계산을 구현했고, JPA Projection으로 조회 컬럼을 최소화했으며 bulkUpsert로 DB I/O를 줄였습니다. 알림 생성 로직은 도메인 이벤트 기반으로 비즈니스 로직에서 분리해, 새로운 이벤트 유형 추가 시 기존 코드를 수정하지 않아도 되는 구조로 구현했습니다. ServiceResult<t> 응답 체계를 설계해 시스템 오류와 비즈니스 실패를 분리했고, 이를 팀 공통 가이드로 만들었습니다. 그 결과, 기능 단위 구현에 머물지 않고 랭킹·알림·예외 처리 전반에 걸쳐 운영 기준이 정리되었고, 백엔드 코드 작성 방식이 프로젝트 후반까지 일관되게 유지됐습니다. 해당 프로젝트는 SSAFY 공통 프로젝트 우수상을 받았습니다.

SSAFY FORCE에서는 팀 가입 요청 동시 처리 시 발생할 수 있는 데이터 정합성 문제를 JPA @Version 기반 낙관적 락으로 해결했습니다. 충돌 빈도가 낮은 환경에서 비관적 락의 불필요한 성능 저하를 피하기 위한 선택이었습니다.

세 프로젝트 모두에서 API 명세 작성, 예외 처리 기준 문서화, 변경 이력 관리를 수행했습니다. 기능 구현만이 아니라 팀이 같은 기준으로 작업하고, 운영 중에 발생한 문제를 빠르게 추적할 수 있는 환경을 만드는 것도 백엔드 개발자의 역할이라는 것을 깨달았습니다.
```

### 5. 장점 / 단점

```
장점 — "기준을 먼저 잡는다"
저는 어떤 작업이든 시작 전에 기준을 먼저 세우는 편입니다. 감각이나 기억에 의존하는 것을 불편하게 여기고, 규칙과 구조로 정리하는 방식을 선호합니다.

TMI 프로젝트에서 팀 내 API 응답 구조가 일관되지 않은 상황을 발견했을 때, 저는 바로 코드를 수정하지 않았습니다. 먼저 "왜 이 구조가 문제인지"를 팀에 설명하고, 비즈니스 실패와 시스템 예외를 구분하는 ServiceResult 응답 체계를 설계한 뒤, 응답 규칙·로깅 기준·API 확장 방법을 문서로 정리해 공유했습니다. 기준이 생기고 나서야 코드를 바꿨습니다.

삼성전자 연계 프로젝트에서도 같은 방식으로 일했습니다. 다중 사용자 환경에서 발생할 수 있는 충돌 문제를 코드로 해결하기 전에, 버전 단위 상태 분리라는 규칙을 먼저 정의하고 ERD에 반영했습니다. 예외 처리 기준을 팀 공통 문서로 먼저 정리했습니다.

이 성향 덕분에 팀원들이 같은 기준 위에서 작업할 수 있었고, 기능 추가 시 커뮤니케이션 비용이 줄었으며, 운영 중에 발생한 문제를 빠르게 원인 추적할 수 있는 환경이 만들어졌습니다.

단점 — "속도보다 구조를 먼저 챙긴다"
기준을 먼저 세우는 성향의 이면은, 초반 속도가 느리다는 점입니다.

저는 새로운 기능을 구현하기 전에 예외 케이스를 먼저 나열하거나, API 명세를 작성하고 나서 코드를 작성하는 순서를 선호합니다. 대부분의 상황에서 이 방식이 결과적으로 더 빠르게 이어졌지만, 일정이 촉박한 구간에서는 팀 전체의 속도를 늦추는 요인이 됐습니다.

삼성전자 연계 프로젝트 중반, 기능 추가 요청이 집중되던 시기에 이 문제가 실제로 발생했습니다. 저는 API 명세를 먼저 정리하고 구현하려 했고, 프론트엔드 팀은 구현 완료를 기다리는 상황이 겹쳤습니다. 그 경험 이후 저는 두 가지 기준을 스스로 정했습니다. 문서화가 필요한 작업과 바로 구현해도 되는 작업을 일정 기준에 따라 구분할 것, 그리고 명세 작성과 구현을 완전히 분리하지 않고 최소 인터페이스만 먼저 공유할 것. 이후 프로젝트에서는 초반에 API 스펙 초안만 먼저 공유하고, 세부 규칙은 구현과 함께 완성하는 방식으로 조정했습니다.
```
